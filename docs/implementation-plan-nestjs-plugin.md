# NestJS Plugin Implementation Plan

> Consensus output from 4-agent architecture debate. Synthesized from competing proposals by Architecture Lead (minimal MVP), Feature Architect (fuller implementation), DX Engineer (developer experience), and Devil's Advocate (stress-testing).
>
> **Status: Implemented.** This document reflects the final decisions and actual implementation.

## Executive Summary

Generate type-safe controller method types from OpenAPI specs for NestJS applications. The plugin generates **per-tag `ControllerMethods` type aliases** mapping operation IDs to typed method signatures. NestJS developers add `implements Pick<PetsControllerMethods, ...>` to their controller classes for compile-time contract enforcement.

**Why per-tag grouping (not flat)?** Tag-based grouping was originally deferred to v1.1, but was implemented directly because it maps naturally to NestJS's one-controller-per-resource pattern. Operations are grouped by their first OpenAPI tag into `{PascalTag}ControllerMethods` types. Untagged operations go to `DefaultControllerMethods`.

| Decorator generation   | Deferred to v2                                            | All agreed: not MVP scope                                                                                              |
| Config options         | None for MVP                                              | Arch Lead: zero config = fastest path to value                                                                         |

## Critical Design Decision: The `implements` Constraint

The Devil's Advocate identified the most important technical finding:

**TypeScript `implements` works with NestJS decorated parameters**, but only if the user adopts whole-object parameter style:

```typescript
// WORKS with implements - whole-object style (recommended)
@Get(':petId')
async showPetById(@Param() path: ShowPetByIdData['path']): Promise<ShowPetByIdResponse> { ... }

// DOES NOT WORK with implements - decomposed style
@Get(':petId')
async showPetById(@Param('petId') petId: string): Promise<ShowPetByIdResponse> { ... }
```

The whole-object style is a recognized NestJS pattern and arguably cleaner. This constraint is prominently documented.

**Known limitations (documented):**

- Methods using `@Res()` for raw response access are incompatible (extra parameter breaks assignability)
- File upload operations using `@UploadedFile()` may need to bypass `implements` for specific methods
- SSE endpoints returning `Observable<MessageEvent>` don't match `Promise<T>` return type

## Actual Output Format

### Generated `nestjs.gen.ts`

For the Petstore spec (with CRUD operations):

```typescript
// This file is auto-generated by @hey-api/openapi-ts

import type {
  CreatePetData,
  CreatePetResponse,
  DeletePetData,
  DeletePetResponse,
  GetInventoryResponse,
  ListPetsData,
  ListPetsResponse,
  ShowPetByIdData,
  ShowPetByIdResponse,
  UpdatePetData,
  UpdatePetResponse,
} from './types.gen';

export type PetsControllerMethods = {
  createPet: (body: CreatePetData['body']) => Promise<CreatePetResponse>;
  deletePet: (path: DeletePetData['path']) => Promise<DeletePetResponse>;
  listPets: (query?: ListPetsData['query']) => Promise<ListPetsResponse>;
  showPetById: (path: ShowPetByIdData['path']) => Promise<ShowPetByIdResponse>;
  updatePet: (
    path: UpdatePetData['path'],
    body: UpdatePetData['body'],
  ) => Promise<UpdatePetResponse>;
};

export type StoreControllerMethods = {
  getInventory: () => Promise<GetInventoryResponse>;
};
```

### Design Decisions in Output

1. **`type` alias (not `interface`)** -- DSL naturally produces `type` aliases; both work with `implements` in TypeScript
2. **Per-tag grouping** -- operations grouped by first OpenAPI tag into `{PascalTag}ControllerMethods`
3. **Individual method parameters** -- `path`, `query`, `body`, `headers` matching NestJS decorator extraction
4. **Required-before-optional sorting** -- params sorted so required come first, preventing TS signature errors
5. **Optional params use `?`** -- uses `hasParameterGroupObjectRequired()` to determine optionality
6. **Return type is `Promise<OperationResponse>`** -- union of success response body types, not status-code-indexed map
7. **No external imports** -- unlike Fastify which imports `RouteHandler` from the `fastify` package, NestJS output is pure TypeScript. Zero runtime dependencies
8. **Untagged operations** -- grouped under `DefaultControllerMethods`

### Parameter Mapping

```
OpenAPI                    → Method Parameter
─────────────────────────────────────────────
operation.parameters.path  → path: OperationData['path']
operation.parameters.query → query?: OperationData['query']
operation.body             → body: OperationData['body']
operation.parameters.header→ headers?: OperationData['headers']
```

Parameters are collected then sorted: required before optional. This prevents TypeScript errors where an optional parameter would precede a required one (e.g., `(query?: T, body: U)` is invalid).

### Response Type Mapping

Uses `operationResponsesMap()` to check for responses, then queries the `role: 'response'` symbol (the union of success response bodies), NOT the `role: 'responses'` status-code-indexed type. Wraps in `Promise<T>`, falls back to `Promise<void>` when no response symbol exists.

## Usage Patterns

### Basic Usage

```typescript
import { Controller, Get, Post, Param, Query, Body } from '@nestjs/common';
import type { PetsControllerMethods } from '../client/nestjs.gen';
import type { ListPetsData, ShowPetByIdData, CreatePetData } from '../client/types.gen';

@Controller('pets')
export class PetsController implements Pick<
  PetsControllerMethods,
  'listPets' | 'createPet' | 'showPetById'
> {
  @Get()
  async listPets(@Query() query?: ListPetsData['query']) {
    return [];
  }

  @Post()
  async createPet(@Body() body: CreatePetData['body']) {
    return { id: 1, name: body.name };
  }

  @Get(':petId')
  async showPetById(@Param() path: ShowPetByIdData['path']) {
    return { id: Number(path.petId), name: 'Kitty' };
  }
}
```

### Incremental Adoption (DX Engineer's Key Insight)

Unlike Fastify where you must provide ALL handlers at once, NestJS developers can adopt incrementally:

```typescript
// Step 1: Add implements to ONE controller
@Controller('pets')
export class PetsController implements Pick<PetsControllerMethods, 'listPets'> {
  @Get()
  async listPets(@Query() query?: ListPetsData['query']) { ... }

  // Other methods remain untyped for now
  @Post()
  async createPet(@Body() body: any) { ... }
}

// Step 2: Gradually add more methods to the Pick union
// TypeScript catches any signature mismatches immediately
```

### Migration from Existing App

```typescript
// BEFORE: existing controller, no generated types
@Controller('pets')
export class PetsController {
  @Get()
  async findAll(@Query('limit') limit?: number) { ... }
}

// AFTER: add implements, rename method to match operationId, use whole-object params
@Controller('pets')
export class PetsController implements Pick<PetsControllerMethods, 'listPets'> {
  @Get()
  async listPets(@Query() query?: ListPetsData['query']) { ... }
}
```

### `plugin.ts` (Core Logic)

The handler groups operations by first tag, builds method signatures per operation, and emits one type alias per tag:

```typescript
export const handler: NestJSPlugin['Handler'] = ({ plugin }) => {
  const operationsByTag = new Map<string, Array<{ name: string; type: ... }>>();

  plugin.forEach('operation', ({ operation, tags }) => {
    const tag = tags?.[0] ?? 'default';
    // ... collect methods per tag
  }, { order: 'declarations' });

  for (const [tag, methods] of operationsByTag) {
    const pascalTag = toCase(tag, 'PascalCase');
    // emit: export type {PascalTag}ControllerMethods = { ... }
  }
};
```

Key implementation details:

- `operationToMethod()` builds each method's function type
- Parameters collected into array, then sorted (required before optional)
- Uses `plugin.querySymbol({ role: 'data' })` for param types
- Uses `plugin.querySymbol({ role: 'response' })` for return types
- Uses `operationResponsesMap()` to verify responses exist before referencing response symbol

## Edge Cases Handled

| Edge Case                   | Handling                                                           | Source                               |
| --------------------------- | ------------------------------------------------------------------ | ------------------------------------ |
| No operationId              | System auto-generates from method+path (e.g., `getPetsByPetId`)    | Already handled by `operationToId()` |
| Duplicate operation IDs     | Numeric suffix appended (`result2`, `result3`)                     | Already handled by `operationToId()` |
| Invalid TS identifier chars | Sanitized by `sanitizeNamespaceIdentifier()`                       | Already handled                      |
| No parameters               | Method has no params: `getInventory: () => Promise<...>`           | Natural                              |
| No response body (204)      | Returns `Promise<void>`                                            | Handled via fallback                 |
| No tags                     | Grouped under `DefaultControllerMethods`                           | Handled                              |
| Multiple tags               | First tag wins                                                     | Handled                              |
| File uploads (multipart)    | Body type is generated but may not match `@UploadedFile()` pattern | Document as limitation               |
| SSE/streaming               | Return type won't match `Observable<MessageEvent>`                 | Document as limitation               |
| `@Res()` usage              | Extra param breaks `implements` assignability                      | Document as limitation               |

## Resolved Open Questions

1. **`interface` vs `type` in DSL** -- `$.type.alias()` produces a `type` alias. Both `type` and `interface` work with `implements` in TypeScript, so this is fine.

2. **Response type symbol** -- `role: 'response'` (singular) returns the flattened union of success response bodies. Combined with `operationResponsesMap()` check to verify responses exist before referencing the symbol.

3. **`$.type.func()` parameter syntax** -- Uses `funcType.param(key, (p) => p.required(...).type(...))` pattern. Verified by examining DSL usage in other plugins.
